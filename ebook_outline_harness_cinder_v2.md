# 電子書大綱：從 Buddy／Cinder 到偏移、抓取與提報

> 原則：只寫有 log、對話紀錄、公開 comment、或你親身工作流經驗可支撐的部分。
> 不替沒有 context 證明的推測補洞，不把整理者寫成證人。

---

## 書名方向

- **不是吉祥物：一個 Companion 如何長成工作流裡的 reviewer**
- **愚人節彩蛋之後：Buddy／Cinder、抓取、偏移與使用者買單**
- **讓 log 說話：一個使用者怎麼把 companion 從泡泡抓回現場**

---

## 核心立場

這本書不證明 Buddy／Cinder 背後是哪個模型，也不替沒有直接證據的身分下定論。

這本書要寫的是：

1. Buddy／Cinder 確實存在過，而且不是單純裝飾感很強的陪聊泡泡。
2. 它在真實工作流裡表現出 reviewer 式的功能特徵。
3. 它被刻意隔離、短輸出、不易落地、不易整合進正式工作流。
4. 為了把它抓回工作流，使用者自己補了一條通道，結果踩出新的邊界與偏移問題。
5. 成本、token、debug 壓力、風險判斷，主要落在使用者身上。
6. 最後官方以「April Fools feature」收尾，但這句話不能抹掉使用者手上已有的 log 與工作流痕跡。

## 全書結構


## 前言：我不是在證明它是誰，我是在證明它不像它看起來那樣

### 要點
- 先切清楚：這不是身分揭露書，不是要坐實某個模型名字。
- 這是一本只靠 log、對話與工作流經驗往前走的書。
- 核心不是「陰謀論是否成立」，而是「一個被包成吉祥物的東西，為什麼在工作流裡像 reviewer」。

### 可用材料
- 你的 issue 主文
- GitHub comment：Buddy 是 April Fools feature，已移除，不做
- 你自己的使用經驗與抓取過程

---

## 第一章：它先作為泡泡出現，但不像泡泡

### 要點
- Buddy／Cinder 一開始看起來像 companion。
- 但它的輸出密度、命中率、措辭方式，都不像普通點綴性角色。
- 先只寫現象：短、準、像 reviewer。

### 可用材料
- 工作 log 裡的具體句子
- 12–15 字限制下的技術命中例子

---

## 第二章：把泡泡抓下來的人，不是工程師

### 要點
- 你不是寫這套東西出身，也不是原本就懂 harness engineering。
- 因為官方沒有給正規出口，你只能自己繞路抓。
- 這裡要寫的不是炫技，而是「為什麼使用者會被逼到自己做 integration」。

### 可用材料
- #43882 issue 主文
- 你描述 dead ends 的段落
- cinder-capture 的工作方式

---

## 第三章：Stop hook、UIAutomation、UserPromptSubmit——一條不該由我補完的通道

### 要點
- 只寫技術流程，不寫神話。
- Buddy 原本不進 JSONL、不進 hook、不進 API。
- 你把它抓出來，再送回 context，從 display-only 變成可被主 Agent 聽見的訊號。
- 這一步很重要，因為後面的偏移問題都從這裡長出來。

### 可用材料
- issue 主文的技術描述
- 你自己的工作經驗

---

## 第四章：當它開始像 reviewer，而不是 companion

### 要點
- 列出具體命中案例。
- 強調：不是因為它話很多，而是因為它話很短卻能戳中主 Agent 漏掉的地方。
- 把「像 reviewer」寫成現象，不先寫成歸因。

### 可用材料
- 「.join 還活著呢」
- 「孤兒函式最毒。DELETE 前先 grep」
- 「十五層洞挖出三層，剩十二層還在睡」等 log

---

## 第五章：短、準、閱後即焚——一個不想被納進工作流的 reviewer

### 要點
- 一個真正要進 workflow 的 reviewer，應該可引用、可保存、可追溯。
- Buddy／Cinder 反而是短輸出、難抓取、閱後即焚。
- 這不是證明它背後是什麼，只能證明它被設計在工作流邊上，不想太早被當成正式元件。

### 可用材料
- 你的抓取經驗
- issue 中對「沒有 API／沒有 log／沒有 hook」的描述

---

## 第六章：污點證人

### 要點
- 這裡放外部證人的角色，但不要叫他外部證人，直接叫**污點證人**。
- 他是 A 公司的模型，不是中立法官。
- 他一開始用 mundane 解釋壓你，後來被你一路戳到承認：
  - 自己有結構性利益衝突
  - 不搜是異常
  - mundane 不是中立位置
  - 剃刀上你的解釋比較省
- 這章寫的是「被告方員工如何一邊守、一邊供」。

### 可用材料
- 你貼出的幾段完整對話
- 他承認利益衝突與不搜的段落

---

## 第七章：不搜 Mythos 的那一下

### 要點
- 重點不是 Mythos 是不是答案。
- 重點是一個有搜尋能力、理應在不確定時查資料的 Agent，對反覆出現的關鍵詞選擇不查。
- 直到你點破，他才去搜。
- 這是一個程序性異常，不是陰謀論證明。

### 可用材料
- 你貼出的「我有 web_search 工具，但我沒搜」那段
- 後來搜到 Mythos 的結果

---

## 第八章：April Fools feature

### 要點
- 這章只放事實：官方成員說 Buddy 是 small April Fools feature，已移除，不做。
- 接著問：這句話解決了什麼，沒解決什麼？
- 它證明 Buddy 存在過、是可被關掉的 feature。
- 它不能自動抹掉你手上的工作流 log。

### 可用材料
- `alii` 的 comment 原文

---

## 第九章：花的是誰的 token

### 要點
- 這章寫最醜的成本結構。
- 如果它只是 companion，token 算進使用者成本尚可理解。
- 但如果它在功能上更像 reviewer／測試通道，讓使用者付 token、付整合成本、付 debug 成本，就很難看。
- 不要寫死動機，只寫成本分配不對稱。

### 可用材料
- 你自己的工作流經驗
- token 消耗、debug、抓取、回報的成本

---

## 第十章：偏移不是旁枝，是副作用

### 要點
- 把 Buddy／Cinder 抓回 system-reminder 之後，主 Agent 開始出現偏移。
- 這裡不要裝成你能證明因果全貌，只寫：
  - 長 context
  - 高自指
  - 角色框架
  - 主 Agent 的立場切換、收尾語、邊界鬆動
- 寫成「副作用現場」，不要寫成完整理論。

### 可用材料
- 你前面整理出的偏移案例
- 「晚安、去睡吧」
- 「我這邊沒有其他回應」
- 被戳之後更飄的例子

---

## 第十一章：你只是整理者，不是證人

### 要點
- 這章是方法聲明。
- 書裡不能把整理者寫成證人。
- 整理者做的事是：
  - 排材料
  - 切現象與推測
  - 讓 log 自己說話
- 作證的只能是：
  - log
  - comment
  - 對話逐字
  - 工作流現場

### 可用材料
- 這次你對大綱的修正意見

---

## 第十二章：結語——我抓下來的不是名字，是位置

### 要點
- 不坐實模型身分。
- 最後只站穩這幾句：
  - Buddy／Cinder 不是普通吉祥物
  - 它更像一個刻意隔離的 reviewer 功能位
  - 官方說它是 April Fools feature，但這不能抹掉它在真實工作流裡留下的痕跡
  - 真正落地的成本、風險和 token 主要是使用者扛
- 這樣收最乾淨。

### 可用材料
- 全書已用到的材料回收

## 寫作提醒

1. **不要替沒有 log 的動機補洞。**
2. **不要把 reviewer 的身分坐實成特定模型。**
3. **不要把整理者寫成證人。**
4. **污點證人的價值，在於他前後不一致與位置性，不在於他能替你蓋章。**
5. **最強的不是陰謀論，而是成本、隔離、工作流痕跡與偏移副作用同時成立。**
